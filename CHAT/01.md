## 三、通信协议设计（B 的核心内容）

在没有硬件的时候，**设计通信协议是你现在就能做的最高价值工作**。协议设计好了，后面编码就是填空。

### 3.1 协议分层

```
应用层协议 (O&M Application Protocol)
  ├── 管理类：发现请求/响应、配对请求/响应、心跳/保活
  ├── 控制类：云台控制、参数查询、参数设置、重启
  └── 数据类：视频流推送、抓拍图片传输
       │
       ▼
传输层 (基于星闪SLE)
  └── 面向连接的数据传输 (类似TCP的概念，但由星闪SLE保证)
```

### 3.2 数据包通用格式（建议）

```
┌─────────────────────────────────────────────────┐
│ 字节偏移    │ 字段          │ 长度    │ 说明     │
├─────────────────────────────────────────────────┤
│ 0          │ 帧头标志      │ 2B     │ 固定 0xAA55 │
│ 2          │ 协议版本      │ 1B     │ 如 0x01   │
│ 3          │ 报文类型      │ 1B     │ 管理/控制/数据 │
│ 4          │ 命令字        │ 2B     │ 具体指令ID  │
│ 6          │ 序列号        │ 2B     │ 用于请求-应答配对 │
│ 8          │ 数据长度      │ 2B     │ Payload 长度 │
│ 10         │ Payload       │ 可变   │ 具体数据    │
│ N          │ 校验和        │ 2B     │ CRC16     │
│ N+2        │ 帧尾标志      │ 2B     │ 固定 0xEB90 │
└─────────────────────────────────────────────────┘
```

## 四、星闪SLE技术原理深潜（A 的核心内容）

### 4.1 星闪 vs 蓝牙 vs WiFi 的关键差异

| 特性 | 星闪 SLE | 蓝牙 BLE | WiFi |
|------|---------|---------|------|
| 最大速率 | **12 Mbps** | 2 Mbps | 300+ Mbps |
| 空口时延 | **0.1ms** | 3-6ms | 10-50ms |
| 连接数 | **256** | 7 (Piconet) | 32/AP |
| 功耗 | 低 | 极低 | 高 |
| 抗干扰 | **强（ Polar码）** | 中（GFSK） | 弱（OFDM） |
| 距离 | 50-100m | 10-30m | 100m |

**SLE 全称 "Sparse-code Low-power Extension"**（稀疏码低功耗扩展），是星闪的一种通信模式，优化了低功耗+中高吞吐场景。

### 4.2 SLE 的工作流程（类比理解）

星闪SLE的连接流程类似于**蓝牙GATT的升级版**：

```
发现阶段： 移动端扫描广播 → 摄像机广播自身信息
   ↓
连接阶段： 移动端发起连接请求 → 摄像机接受 → 连接建立
   ↓
服务发现： 移动端查询摄像机的SLE Server支持哪些服务/特征
   ↓
数据传输： 读写特征值 / 通过Notification接收数据
   ↓
断开阶段： 任一方发起断开
```

对于视频这种大数据量传输，SLE通常提供两种模式：
- **面向连接的数据传输（CO）**：可靠传输，适合控制指令
- **无连接的数据传输（CL）**：低开销，适合视频流（允许少量丢包）

### 4.3 摄像机端软件架构（关键！）

摄像机端的软件架构大致如下：

```
┌─────────────────────────────────────────┐
│  用户态                                  │
│  ┌───────────────────────────────────┐  │
│  │ O&M Service Daemon (运维服务守护进程) │  │
│  │  ├─ SLE 通信管理模块               │  │
│  │  ├─ 指令解析/执行引擎               │  │
│  │  ├─ 视频流推送服务                  │  │
│  │  └─ 参数管理模块                    │  │
│  ├───────────────────────────────────┤  │
│  │ 摄像机 HAL (硬件抽象层)             │  │
│  │  ├─ Camera HAL (视频采集)          │  │
│  │  ├─ PTZ HAL (云台控制)             │  │
│  │  └─ Sensor HAL (传感器)            │  │
│  └───────────────────────────────────┘  │
├─────────────────────────────────────────┤
│  内核态                                  │
│  ┌───────────────────────────────────┐  │
│  │ 星闪模组驱动 (UART/SPI/PCIe)      │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

**关键点**：你写的 `O&M Service Daemon` 是一个 Linux 用户态守护进程。它通过 UART（串口）与星闪模组通信（发AT指令或调用SDK API），通过电鸿的HAL接口控制摄像头和云台。**你不需要碰内核！**
