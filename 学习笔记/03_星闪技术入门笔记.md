## 一、星闪到底是什么？

**星闪（NearLink）** 是咱们中国自己搞的新一代无线短距通信技术。

类比理解：

```
蓝牙像  ──→ 自行车：省力（低功耗），但骑不快（带宽小），只能带少量东西
WiFi像  ──→ 汽车：跑得快（带宽大），但费油（功耗高），在复杂路况容易抛锚
星闪像  ──→ 电动摩托：兼具省电（低功耗）和跑得快（高带宽），路况适应性强
```

**赛题用星闪的原因：**

| 需求 | 蓝牙 | WiFi | 星闪 |
|------|------|------|------|
| 传视频（720P） | ❌ 带宽不够 | ✅ 可以 | ✅ 12Mbps 够用 |
| 低延迟控制 | ✅ 可以 | ❌ 延迟高 | ✅ 0.1ms |
| 工业环境抗干扰 | ❌ 弱 | ❌ 弱 | ✅ 强 |
| 电网安全 | ✅ 安全 | ❌ 有隐患 | ✅ 加密+安全 |
| 功耗 | ✅ 低 | ❌ 高 | ✅ 低 |

## 二、星闪的两种模式

星闪有两种工作模式，赛题用的是 **SLE 模式**：

```
星闪技术
  ├── SLE 模式（Sparse-code Low-power Extension）
  │   └── 对标蓝牙：低功耗 + 中高速率（12Mbps）
  │   └── **这是赛题使用的模式**
  │
  └── SLB 模式（SparkLink Basic）
      └── 对标 WiFi：高速率（目标 Gbps），低时延（20μs）
      └── 适合更高速场景，但功耗高，赛题不用
```

> **你的关注点只有 SLE。** 记住三个关键数据就够了：
> - **最大速率 12Mbps** = 每秒最多传 1.5MB 数据
> - **空口时延 0.1ms** = 从发出到对方收到只需 0.1 毫秒
> - **连接距离 ≥50米** = 空旷环境下至少 50 米

## 三、SLE 的连接过程（手机找摄像机）

整个过程分为四步，你可以类比"蓝牙配对"来理解：

### 第1步：广播（谁在附近？）

```
摄像机（服务端）：      "我在这里！我是电鸿摄像机！我的ID是XXX"
                          ↕  不停地发广播包
手机/平板（客户端）：    听到广播 → "哦！有一台摄像机在附近！"
```

> 这一步在代码里的实现：摄像机端调用星闪 SDK 的 `SLE_StartBroadcast()`，手机端调用 `SLE_StartScan()`。范围 10 米内，赛题要求 10 秒内发现。

### 第2步：连接（我对上话了！）

```
手机：    "摄像机你好，我要连你！这是我的连接参数..."
             ↕
摄像机：  "好的，连接已建立！"
```

> 这一步：手机调用 `SLE_Connect()`，摄像机响应后连接建立。赛题要求 **连接时间 ≤ 500ms**（从手机发起连接到建立完成）。

### 第3步：数据传输（开始聊天）

连接建立后，手机和摄像机之间有了一个"数据通道"，可以双向收发数据：

```
手机 ────→ 摄像机：  控制指令（"云台向左转30度"）
手机 ←──── 摄像机：  执行结果（"已执行"）
手机 ←──── 摄像机：  视频数据流（持续发送 H.264 编码的视频帧）
手机 ────→ 摄像机：  心跳包（"我还活着"）
手机 ←──── 摄像机：  心跳应答
```

> SLE 提供两种数据传输方式：
> - **CO（面向连接）**：可靠传输，类似 TCP。适合控制指令——必须确保对方收到
> - **CL（无连接）**：低开销，有丢包可能。适合视频流——偶尔丢一帧看不出来，重传反而增加延迟

### 第4步：断开（说再见）

```
手机：    "我要断开了，再见"
手机/摄像机 任一方都可以发起断开
```

## 四、物理连接方式：星闪模组怎么和主控通信

**星闪模组是一个独立的小硬件**，通过物理接口和主控芯片（摄像机的主芯片）连接：

```
摄像机主控芯片（电鸿系统）
   │
   ├── UART 串口（最常见） ←── 接星闪模组
   │   └── 速率：921600 bps 或更高
   │   └── 你通过 /dev/ttyS0 或 /dev/ttyUSB0 收发数据
   │
   ├── SPI 接口（较少见，但更快）
   │   └── 适合用 SPI 传输视频数据
   │
   └── USB 接口（移动端常用）
       └── 手机通过 USB 外接星闪模组
```

**在摄像机端，你的代码和星闪模组的交互方式：**

```
你的 C 程序
  ↓  write(fd, at_command, len)    发 AT 指令
UART 串口
  ↓
星闪模组
  ↓  转换成无线信号发射
另一端的星闪模组
```

> 💡 **知识卡片：什么是 AT 指令？**
> AT 指令是一种"文本命令"，起源于拨号上网时代。格式很简单：`AT+命令=参数`，返回 `OK` 或 `+命令: 返回值`。
>
> 例：发送 `AT+SCAN=1` → 开始扫描设备 → 收到 `+SCAN: "Camera-001", -65dBm` 表示发现了一个设备
>
> 厂商通常会把星闪的操作封装成 AT 指令，你的 C 程序只需要读写串口发 AT 指令就行了。

## 五、移动端（手机/平板）的星闪连接

手机的情况比较特殊——**目前几乎没有手机自带星闪芯片**。

你的方案有两种：

### 方案 A：USB 外接星闪模组（推荐）

```
手机/平板
  └── USB Type-C 接口
       └── 外接星闪模组（USB 转 UART 或直接 USB）
            └── 通过 Android 的 USB Serial API 读写数据
```

你的 Android APP 通过 USB 接口连接星闪模组，发送和接收数据的流程：

```java
// 伪代码，理解流程
UsbDevice sparkModule = findSparkModule();        // 找到星闪模组
UsbDeviceConnection conn = connect(sparkModule);   // 建立 USB 连接
conn.bulkTransfer(endpoint, data, len, timeout);   // 发送数据到星闪模组
conn.bulkTransfer(endpoint, buffer, len, timeout); // 从星闪模组读取数据
```

### 方案 B：外接开发板桥接

```
手机 APP ←→ 微小型开发板（如 ESP32 作为桥接器） ←→ 星闪模组
```

> **实际操作建议：** 你拿到的硬件套件中，移动端应该会提供 USB 接口的星闪模组。先用方案 A，后面有时间再优化。

## 七、一句话总结

> **摄像机通过 UART 串口给星闪模组发 AT 指令，星闪模组把指令转成无线电波发出去；手机通过 USB 接口接上星闪模组，收到电波后转成数据传给 APP。** 你的工作就是写这两端的数据收发和业务逻辑代码。
